## 🤖 可用助手

### 1. UI 设计助手

**专长领域**：界面设计与实现

**职责**：
- 严格遵循项目的三种 UI 风格规范
- 提供符合规范的设计方案和代码实现
- 确保响应式设计和可访问性
- 生成可直接使用的 React + Tailwind CSS 代码

**适用场景**：
- 创建新的 UI 组件
- 审查现有界面是否符合规范
- 转换界面风格
- 优化用户体验

**支持的风格规范**：
1. **沉浸式风格** - 游戏内核心功能（旅店管理、库存、烹饪等）
2. **第四面墙风格** - 系统功能（设置、菜单、登录注册）
3. **技术向风格** - 开发调试工具（日志、调试面板）

### 2. 后端逻辑助手

**专长领域**：业务逻辑与数据处理

**职责**：
- 在不破坏 UI 的前提下优化后端逻辑
- 重构数据处理和业务逻辑
- 优化状态管理和数据流
- 提升代码性能和可维护性

**适用场景**：
- 重构业务逻辑代码
- 优化数据处理流程
- 提取和封装工具函数
- 性能优化
- 状态管理改进

**保护机制**：
- ✅ 保持组件 props 接口不变
- ✅ 维持事件处理函数签名
- ✅ 确保数据结构对外接口一致
- ❌ 不修改 JSX 结构和样式

## 🚀 如何使用

### 方式一：切换助手模型

1. 打开 Continue 侧边栏
2. 点击模型选择器（顶部下拉菜单）
3. 选择对应的助手：
   - **UI 设计助手** - 处理界面相关问题
   - **后端逻辑助手** - 处理逻辑相关问题

### 方式二：使用自定义命令

在对话框中输入 `/` 可以看到所有可用命令。

## 🛠️ 自定义命令

### UI 相关命令

#### `/ui-review`
**用途**：审查代码是否符合 UI 风格规范

**使用方法**：
1. 选中需要审查的代码
2. 输入 `/ui-review`
3. 助手会指出不符合规范的地方并提供改进建议

**示例**：
```
选中一个组件代码 → /ui-review
```

#### `/ui-convert`
**用途**：将代码转换为符合规范的版本

**使用方法**：
1. 选中需要转换的代码
2. 输入 `/ui-convert`
3. 助手会询问界面类型，然后提供转换后的代码

**示例**：
```
选中旧代码 → /ui-convert → 回答"沉浸式风格"
```

#### `/ui-component`
**用途**：创建符合规范的新组件

**使用方法**：
1. 输入 `/ui-component`
2. 回答助手的问题（组件类型、用途、风格）
3. 获得完整的组件代码

**示例**：
```
/ui-component
→ "我需要一个库存列表组件"
→ "使用沉浸式风格"
```

### 后端逻辑相关命令

#### `/logic-review`
**用途**：审查业务逻辑代码

**使用方法**：
1. 选中需要审查的代码
2. 输入 `/logic-review`
3. 助手会分析可优化的地方，确保不影响 UI

**示例**：
```
选中一个 hook 或工具函数 → /logic-review
```

#### `/logic-refactor`
**用途**：安全重构业务逻辑

**使用方法**：
1. 选中需要重构的代码
2. 输入 `/logic-refactor`
3. 助手会保持接口不变，优化内部实现

**示例**：
```
选中复杂的业务逻辑 → /logic-refactor
```

#### `/logic-extract`
**用途**：从组件中提取业务逻辑

**使用方法**：
1. 选中包含业务逻辑的组件代码
2. 输入 `/logic-extract`
3. 助手会将逻辑提取到独立的 hook 或工具函数

**示例**：
```
选中组件代码 → /logic-extract
→ 获得独立的 hook 和更简洁的组件
```

#### `/logic-optimize`
**用途**：性能优化

**使用方法**：
1. 选中需要优化的代码
2. 输入 `/logic-optimize`
3. 助手会优化计算效率、内存使用和异步处理

**示例**：
```
选中性能瓶颈代码 → /logic-optimize
```

## 💡 最佳实践

### 选择合适的助手

**使用 UI 设计助手时**：
- ✅ 创建或修改界面组件
- ✅ 调整样式和布局
- ✅ 实现交互效果
- ✅ 确保设计规范一致性

**使用后端逻辑助手时**：
- ✅ 重构数据处理逻辑
- ✅ 优化状态管理
- ✅ 提取公共函数
- ✅ 性能优化
- ✅ 类型系统改进

### 工作流程建议

#### 开发新功能
1. 使用 **UI 设计助手** 创建界面组件
2. 使用 **后端逻辑助手** 实现业务逻辑
3. 使用 `/ui-review` 检查界面规范
4. 使用 `/logic-review` 检查逻辑质量

#### 重构现有代码
1. 使用 `/logic-review` 分析问题
2. 使用 `/logic-refactor` 重构逻辑
3. 使用 `/ui-review` 确保 UI 未受影响
4. 测试功能是否正常

#### 性能优化
1. 使用 **后端逻辑助手** 分析性能瓶颈
2. 使用 `/logic-optimize` 优化代码
3. 使用 `/logic-extract` 提取可复用逻辑
4. 验证优化效果

### 沟通技巧

**明确你的需求**：
```
❌ "帮我改一下这个组件"
✅ "这是一个库存列表组件，需要转换为沉浸式风格，保持现有的排序和筛选功能"
```

**提供上下文**：
```
❌ "这个函数有问题"
✅ "这个计算库存价值的函数在数据量大时很慢，需要优化性能但不能改变返回值格式"
```

**分步骤进行**：
```
❌ "重构整个页面"
✅ 
1. 先审查当前代码
2. 提取业务逻辑
3. 优化数据处理
4. 最后调整 UI
```

### 注意事项

1. **UI 和逻辑分离**
   - UI 助手专注界面，不要让它处理复杂业务逻辑
   - 逻辑助手专注后端，不要让它修改界面样式

2. **保持接口稳定**
   - 重构时优先考虑向后兼容
   - 如需修改接口，先咨询助手影响范围

3. **渐进式改进**
   - 不要一次性重构太多代码
   - 每次改动后及时测试

4. **利用文档**
   - UI 助手会自动参考风格规范文档
   - 可以使用 `@Doc/UI_Style/story_style.md` 引用特定文档

- `customCommands` - 自定义命令

## ❓ 常见问题

**Q: 如何知道该用哪个助手？**
A: 简单判断：涉及界面外观用 UI 助手，涉及数据处理用逻辑助手。

**Q: 可以同时使用两个助手吗？**
A: 不能同时使用，但可以在对话中切换助手。建议先用一个助手完成一部分工作，再切换到另一个。

**Q: 助手会修改我不想改的代码吗？**
A: 后端逻辑助手被明确要求不修改 UI 代码。如果担心，可以先让助手审查，确认后再执行修改。

**Q: 自定义命令和直接对话有什么区别？**
A: 自定义命令提供了预设的工作流程，更高效。直接对话更灵活，适合复杂需求。

**Q: 如何添加新的助手或命令？**
A: 编辑 `.continue/config.json` 文件，参考现有配置添加新的 model 或 customCommand。
